Exercice 5 — Préconditions et Postconditions

Intermédiaire ◎ 10 pts ⏱ 10–15 min
Objectifs
  • Comprendre ce que signifie un triplet de Hoare {P} C {Q}
  • Calculer une précondition la plus faible (weakest precondition) par substitution
  • Dériver la postcondition la plus forte à partir d'une précondition donnée

Prérequis : Cours 2 — Sémantique axiomatique, logique de Hoare

De l'exécution à la preuve

Jusqu'ici, la sémantique opérationnelle décrivait comment un programme s'exécute — étape par étape. La sémantique axiomatique (logique de Hoare) change de perspective : elle spécifie ce que fait un programme en termes de propriétés logiques, sans simuler son exécution. On ne demande plus "quelle valeur a x après ?" mais "quelle propriété est garantie après ?".

Le triplet de Hoare

Un triplet de Hoare s'écrit {P} C {Q} et se lit :

"Si la propriété P est vraie avant d'exécuter C, et si C termine, alors la propriété Q est vraie après."

C'est une promesse conditionnelle : si P est fausse au départ, le triplet ne dit rien du tout sur le résultat.

Note : les triplets de Hoare standards expriment la correction partielle — ils supposent que le programme termine. La preuve de terminaison est séparée.

Analogie — Le contrat

Un triplet de Hoare est comme un contrat entre un appelant et une fonction :

  • L'appelant promet : "P sera vraie avant l'appel."
  • La fonction promet : "Si tu respectes P, je garantis Q à la sortie."
  • Si l'appelant viole sa part (P est fausse), la fonction n'a plus aucune obligation.

C'est exactement le principe du Design by Contract (Eiffel, Ada, Dafny).

Analogie — La recette

  • Précondition : "Vous avez 200g de farine, 2 œufs, un four à 180°C."
  • Programme : mélanger, verser, enfourner 25 min.
  • Postcondition : "Vous avez un gâteau cuit."

Si vous démarrez sans œufs (P fausse), la recette ne garantit pas de gâteau. Elle ne dit rien de ce qui se passe alors.

La précondition la plus faible — wp

Plutôt que de vérifier un triplet donné, on peut calculer automatiquement la précondition minimale à partir de la postcondition. C'est le calcul de précondition la plus faible (weakest precondition) :

wp(C, Q) = "la condition minimale à vérifier avant C pour que Q soit vraie après"

On travaille à rebours — de la postcondition vers la précondition.

Analogie : wp, c'est lire le mode d'emploi à l'envers. On part du résultat voulu ("l'appareil fonctionne") et on remonte vers les conditions minimales requises ("alimentation 220V, pas d'immersion dans l'eau"). C'est exactement ce que font les vérificateurs automatiques (Why3, Frama-C, Dafny).

Règle wp pour l'affectation

C'est la règle fondamentale :

wp(x := e, Q) = Q[e/x]

On substitue l'expression e à la place de chaque occurrence de x dans Q.

CommandePostcondition Qwp = Q[e/x]
x := 0x = 00 = 0 → Vrai
x := x + 1x > 0x + 1 > 0 → x > -1
x := y + 1x > 0y + 1 > 0 → y > -1
x := 3x + y > 03 + y > 0 → y > -3
Piège classique : substituer dans le mauvais sens — remplacer e par x au lieu de x par e. La règle est : dans Q, partout où apparaît la variable assignée (x), la remplacer par l'expression du membre droit (e).

Précondition la plus faible vs postcondition la plus forte

Ces deux notions sont duales :

Précondition la plus faible (wp)Postcondition la plus forte (sp)
Direction ← Raisonnement à rebours (de Q vers P) → Raisonnement avant (de P vers Q)
Question Quelle hypothèse minimale avant C garantit Q après ? Quelle propriété maximale est garantie après C, sachant P avant ?
"La plus faible/forte" Impose le moins de contraintes → admet le plus d'états initiaux Décrit le plus précisément les états finaux possibles

Intuition : une condition est d'autant plus forte qu'elle décrit un petit ensemble d'états (x = 5 ∧ y > 0 est plus forte que x = 5). La précondition la plus faible est donc celle qui impose le moins — elle autorise le plus grand ensemble d'états de départ pour lesquels le programme respecte Q.

Précondition Vrai et précondition Faux

  • {Vrai} C {Q} : C garantit Q quelle que soit la valeur initiale des variables — le programme est totalement sûr vis-à-vis de Q.
  • {Faux} C {Q} : toujours valide vacuoirement (aucun état ne satisfait Faux, donc la promesse n'est jamais déclenchée). Valid mais totalement inutile en pratique.

Exemple guidé — wp d'une affectation

Calculer wp(y := y + 2, y > 10).

  1. Postcondition : y > 10
  2. Substituer y + 2 à la place de y : (y + 2) > 10
  3. Simplifier : y > 8

Vérification : si y = 9 (satisfait y > 8), après y := y + 2, y = 11 > 10. ✓

Exemple guidé — postcondition la plus forte

Précondition x > 0, commande x := x * 2.

  1. On sait que x > 0 avant.
  2. Après x := x * 2, la nouvelle valeur est x * 2.
  3. La postcondition la plus forte est ce qu'on peut dire exactement : x > 0 (car si l'ancienne valeur était > 0, la nouvelle vaut le double, donc aussi > 0).

On peut également vérifier par wp : wp(x := x*2, x > 0) = x*2 > 0 = x > 0. Cohérent.

Q1 5 pts Fondamental

Quelle est la précondition la plus faible P pour que le triplet suivant soit prouvable ?

{P} x := 0 {x = 0}

Piège courant : répondre x = 0 ou x ≥ 0 par réflexe. La précondition ne doit pas sur-contraindre l'état initial — cherchez la condition minimale, c'est-à-dire celle qui est vraie pour le plus grand nombre d'états de départ possibles.
Appliquez la règle wp pour l'affectation : wp(x := 0, x = 0). Il faut substituer 0 à la place de x dans la postcondition x = 0. Qu'obtenez-vous ?
En substituant : 0 = 0. Cette proposition est toujours vraie, indépendamment de toute variable. La précondition la plus faible est donc Vrai. Intuitivement : peu importe l'état initial, x := 0 met toujours x à 0 — la postcondition est toujours satisfaite.

Solution — Q1

Application de la règle wp :

wp(x := 0, x = 0)
= substituer 0 à la place de x dans x = 0
= 0 = 0
= Vrai

Interprétation

La précondition Vrai signifie : aucune hypothèse n'est nécessaire sur l'état initial. Quelle que soit la valeur de x avant l'affectation, l'instruction x := 0 écrase cette valeur et place 0 dans x. La postcondition x = 0 est donc toujours satisfaite.

On dit que {Vrai} x := 0 {x = 0} est un triplet universellement valide : il n'impose aucune restriction sur l'appelant.

Toute précondition plus forte (ex. x > 0) est aussi valide, mais elle sur-contraint inutilement : elle exclut des états initiaux parfaitement légitimes.

Q2 5 pts Intermédiaire

Quelle est la postcondition la plus forte Q pour que le triplet suivant soit prouvable ?

{x ≥ 0} x := x + 2 {Q}

Piège courant : répondre Q = Vrai ou Q = x ≥ 0. Ces postconditions sont valides mais pas les plus fortes — elles perdent de l'information. La plus forte est celle qui décrit le plus précisément l'état final, sans rien laisser de flou.
Si x ≥ 0 avant l'affectation, quelle est la valeur minimale que peut atteindre x après x := x + 2 ? Pensez aux cas limites : si x = 0 avant, quelle est la valeur après ?
Si x ≥ 0 avant, la nouvelle valeur de x est x + 2 ≥ 0 + 2 = 2. Donc Q = x ≥ 2. Vérifiez avec wp : wp(x := x+2, x ≥ 2) = x+2 ≥ 2 = x ≥ 0. C'est exactement notre précondition — le calcul boucle cohéremment. ✓

Solution — Q2

Raisonnement avant (de P vers Q) :

  • On sait que x ≥ 0 avant l'affectation.
  • Après x := x + 2, la nouvelle valeur de x est l'ancienne + 2.
  • Donc : nouvelle valeur ≥ 0 + 2 = 2, soit x ≥ 2.

La postcondition la plus forte est Q = x ≥ 2.

Vérification par wp (cohérence)

On peut vérifier que ce Q est correct en calculant wp à rebours :

wp(x := x + 2, x ≥ 2)
= substituer x + 2 à la place de x dans x ≥ 2
= x + 2 ≥ 2
= x ≥ 0

On retrouve exactement la précondition de départ. Le triplet est bien formé.

Pourquoi x ≥ 0 ou Vrai sont-ils insuffisants comme Q ?

Ces postconditions sont valides (le triplet est prouvable), mais elles sont trop faibles : elles disent moins que ce qu'on peut réellement conclure. En pratique, une postcondition trop faible rend la spécification inutile — on ne peut pas s'appuyer dessus pour prouver des propriétés ultérieures. La postcondition la plus forte est celle qui capture tout ce qu'on peut déduire.
