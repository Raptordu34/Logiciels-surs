Chap 3 — Développement de logiciels sûrs

Synthèse

  • Méthode B — approche formelle pour développer des programmes corrects par construction, modélisés par des Machines Abstraites.
  • Machine Abstraite (MA) — encapsule l'état (variables, invariants) et les services (opérations) du système.
  • Obligations de Preuve (PO) — garanties mathématiques : l'initialisation établit l'invariant, chaque opération préserve l'invariant.
  • Raffinement — démarche progressive transformant un modèle abstrait non déterministe en code exécutable déterministe, tout en conservant les preuves.

1. Introduction à la méthode B

Le développement formel vise à produire des logiciels garantis sans bug par construction. La méthode B, très répandue dans l'industrie critique, en est un excellent exemple.

Langage de spécification
Basé sur la théorie des ensembles et la logique du premier ordre pour modéliser les données et les propriétés attendues.
Méthode de développement
Démarche progressive du modèle abstrait de haut niveau vers l'implémentation finale (raffinement successif).
Ligne 14 du Métro parisien (octobre 1998) : Le premier métro automatique a été développé avec la méthode B. Environ 110 000 lignes de modèle B ont généré 86 000 lignes de code Ada. Résultat : aucun bug détecté lors des tests, ni en opération à ce jour.

2. Composants d'une Machine Abstraite (MA)

L'approche B est orientée "modèle à états". Le système est décrit par une machine abstraite contenant un état interne, des propriétés invariantes, une initialisation et des opérations (transitions).

MACHINE Nom (paramètres)
SETS S                     -- Ensembles définissant les types
VARIABLES V                -- Variables d’état
INVARIANTS INV             -- Propriétés d’invariance (ce qui doit toujours être vrai)
INITIALISATION INIT        -- État de départ
OPERATIONS op  P|S        -- Services offerts
END;
Opérations : Spécifier une opération, c'est définir sa précondition (les contraintes d'entrée) et son effet (corps de l'opération modifiant l'état).

3. Exemple : Système de réservation

On modélise un système de réservation de places numérotées de $1$ à $nbmax$.

MACHINE Reservation(nbmax)
CONSTRAINTS
    nbmax : 1..MAXINT
DEFINITIONS
    SIEGES == (1..nbmax)
VARIABLES
    occupes, nblibre
INVARIANT
    occupes ⊆ SIEGES ∧
    nblibre ∈ 0..nbmax ∧
    nblibre = nbmax - card(occupes)
INITIALISATION
    occupes := ∅ || nblibre := nbmax
OPERATIONS
    nb <-- place_libre = BEGIN nb := nblibre END

    pp <-- reserver =
       PRE card(occupes) ≠ nbmax THEN
           ANY place WHERE place ∈ SIEGES - occupes THEN
               pp := place || 
               occupes := occupes ∪ {place} || 
               nblibre := nblibre - 1
           END
       END

    liberer(place) =
       PRE place ∈ occupes THEN
           occupes := occupes - {place} || 
           nblibre := nblibre + 1
       END
END

4. Obligations de Preuve (PO - Proof Obligations)

Pour garantir la correction mathématique du modèle défini par une MA, le développeur doit décharger des preuves appelées Obligations de Preuve (PO).

Soit $I$ l'invariant, $U$ l'initialisation, et $S$ une opération sous la précondition $P$.
  1. Initialisation : L'initialisation doit établir l'invariant. On doit prouver $[U]I$.
  2. Conservation : Chaque opération invoquée dans sa précondition doit préserver l'invariant. On doit prouver $I \land P \implies [S]I$.

Application à l'opération de réservation

Lorsqu'on réserve une place, la post-condition (nouvel état) remplace occupes par occupes ∪ {place} et nblibre par nblibre - 1. Les PO s'écrivent :

$$occupes \cup \{place\} \subseteq SIEGES$$ $$nblibre - 1 \in 0..nbmax$$ $$nblibre - 1 = nbmax - \text{card}(occupes \cup \{place\})$$ Exemple de PO à démontrer à partir de $I \land P$

5. Raffinement et Implémentation

Le raffinement est l'étape où l'on ajoute des précisions au modèle abstrait pour se rapprocher du code exécutable. Il consiste à :

  • Représenter concrètement des données abstraites (ex: remplacer un ensemble mathématique occupes par un tableau de booléens etat).
  • Réduire le non-déterminisme (ex: l'opération reserver utilisait ANY place ; le raffinement va imposer de prendre la première place libre, via min()).
  • Affaiblir les préconditions si nécessaire.
Invariant de liaison : Lors du raffinement, on doit définir un lien entre les variables abstraites et concrètes. Par exemple : $occupes = \{s \in SIEGES \mid etat(s) = \text{TRUE}\}$.

L'implémentation : C'est l'étape ultime du raffinement, où le code respecte toutes les contraintes d'un langage de programmation impératif classique (retrait des affectations parallèles au profit du séquencement, utilisation de pointeurs, etc.).
