Annexe 1 — Machine autoEcoleRamoa

Documentation

Contexte — Machine abstraite B

Cette annexe présente le squelette de la machine abstraite autoEcoleRamoa, référence pour l'exercice 6. Elle modélise un système de gestion d'auto-école : les candidats inscrits, ceux ayant validé l'examen du code, et les titulaires du permis.

La méthode B (Jean-Raymond Abrial, 1996) formalise les systèmes logiciels par des machines abstraites : chaque machine déclare un état (variables), des contraintes invariantes sur cet état, une initialisation et des transitions (opérations).

Les clauses d'une machine B, dans l'ordre :

ClauseRôle
MACHINENom de la machine
SETSEnsembles abstraits — types de base non interprétés
VARIABLESVariables d'état, modifiées par les opérations
INVARIANTContrainte toujours vraie sur les variables (type + propriétés métier)
INITIALISATIONValeur initiale des variables — doit satisfaire l'invariant
OPERATIONSTransitions : chacune comporte une précondition (PRE) et une action (THEN)

Obligation de preuve

Chaque opération doit être prouvée : si l'invariant est vrai avant l'opération et que la précondition est satisfaite, alors l'invariant doit être vrai après l'action. La PRE n'est pas un test exécuté — c'est une obligation que l'appelant garantit.

Explication de chaque clause

SETS — POP

POP représente la population totale des personnes pouvant interagir avec le système. C'est un ensemble abstrait non interprété : la méthode B ne lui donne aucune valeur concrète. Chaque candidat est un élément pp ∈ POP.

L'abstraction de POP permet de raisonner sur le système indépendamment de toute représentation informatique des personnes (numéro, nom, identifiant…).

VARIABLES

VariableSignificationType attendu dans l'invariant
inscrit Ensemble des personnes inscrites à l'auto-école inscrit ⊆ POP
aCode Ensemble des personnes ayant réussi l'examen du code aCode ⊆ inscrit
aPermis Ensemble des personnes titulaires du permis de conduire aPermis ⊆ aCode

Logique métier

Contrainte naturelle : obtenir le permis suppose d'avoir réussi le code, qui suppose d'être inscrit. Ces inclusions en cascade doivent figurer dans l'invariant :

aPermis ⊆ aCode ⊆ inscrit ⊆ POP

Cette chaîne d'inclusions est la propriété de sûreté centrale du système.

INITIALISATION

inscrit, aCode, aPermis := ∅, ∅, ∅

Affectation simultanée des trois variables à l'ensemble vide. Au démarrage du système, personne n'est inscrit, n'a validé le code, ni n'a le permis.

Cette initialisation doit satisfaire l'invariant : ∅ ⊆ ∅ ⊆ ∅ ⊆ POP ✓ — l'ensemble vide est inclus dans tout ensemble.

Sections à compléter dans l'exercice 6 :
  • L'invariant complet, en écrivant explicitement les contraintes de type et d'inclusion entre les trois variables.
  • Les opérations permettant d'inscrire un candidat, de valider le code et d'attribuer le permis — chacune avec sa PRE et son THEN.

Méthode

Structure type d'une opération B :

nom_operation(pp) =
PRE
    pp ∈ POP ∧            -- contrainte de type sur le paramètre
    pp ∉ inscrit           -- précondition métier (état requis avant l'opération)
THEN
    inscrit := inscrit ∪ {pp}   -- action : modification de l'état
END

La clause PRE n'est pas un test conditionnel — c'est une obligation. L'appelant garantit la précondition ; en contrepartie, la machine garantit que l'invariant est préservé après l'action. Toute opération appelée hors de sa précondition est indéfinie.

Exemple

Opération s_inscrire(pp) :

s_inscrire(pp) =
PRE
    pp ∈ POP ∧
    pp ∉ inscrit
THEN
    inscrit := inscrit ∪ {pp}
END

Précondition : pp existe et n'est pas déjà inscrit. Action : on l'ajoute à l'ensemble inscrit. L'invariant inscrit ⊆ POP est préservé car pp ∈ POP. ✓
